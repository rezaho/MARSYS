### Polish + advanced features (each likely its own mini-batch of sessions)

- Workflow comments / annotations

- GPU-local model support beyond what marsys framework's current backends provide

## Strategic gates before starting v0.5

- Evaluate user feedback from v0.3 and v0.4 — which items are actually requested vs imagined
- Re-confirm SP-008 (positioning vs MARSYS Studio) with the user, ESPECIALLY before v0.5-05 (multi-user)
- Decide whether OSS positioning still holds — if Spren is drawing customers away from Studio, business model decision precedes more features

## Hard constraints (carry over)

All v0.3/v0.4 constraints apply: no mocks, no backward-compat, full test coverage, framework purity (SP-018), API-as-single-source-of-truth (SP-019). Plus:

- **Multi-user (v0.5-05) requires a security re-review** — single-user threat model in [`07-security.md`](../../architecture/spren/07-security.md) does not cover multi-user adversaries; need new doc + review.
- **Tunnel-based messengers (v0.5-01, v0.5-02) inherit the v0.4 tunnel + allowlist + confirm-before-acting model.** No reduction in security posture.
- **Hosted-mode (v0.5-04) requires explicit user opt-in** — never default. Default stays localhost-bind.

## When v0.5 even matters

v0.5 is *not* the v0.3 / v0.4 finishing line. v0.3 + v0.4 should stand alone as a complete product for the OSS audience. v0.5 only happens if:
- v0.3 + v0.4 land successfully and have engaged users
- We hear genuine demand for these specific items
- Building them doesn't conflict with the proprietary product line
