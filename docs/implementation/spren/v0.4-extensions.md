# v0.4 — TUI, Tauri Polish, Always-on Org & Channels

## Outcome

A v0.3 user gains:

- **A terminal-resident surface.** `spren tui` opens a four-pane Textual interface (Now / Inbox / Activity / Chat) with cmdk-style `:` palette navigation. Same FastAPI backend as the GUI; same domain rendered in a different chrome.
- **A polished native desktop experience.** Tray icon, native menus, autostart on boot, OS notifications, deep links, file associations (drop a `.py` workflow on the icon), polished auto-updater UI.
- **A real agent organization.** The main agent can spawn working instances of itself for parallel work, supervise them, and integrate their results. The user can create teams (Research, Admin, Personal Life, etc.) — each with a team manager and team-scoped sandbox.
- **A full skills catalog.** Skills (capability bundles) like `workflow-inspection`, `web-research-with-citations`, `debugging`, `planning`, `code-review` are loadable into any sub-instance.
- **Real long-term memory.** The discovery index (Tier 4) ships — hybrid BM25 + vector + reranker over markdown + session log. The `recall` tool gives the agent fuzzy semantic search. Daily consolidation pipeline runs (24h+5-session trigger) by a cheap model.
- **External triggers.** Workflows can be triggered on a cron schedule, by inbound webhook, OR by Telegram / Discord. The `cloudflared` quick-tunnel ships as `spren expose` for webhook-required setups.
- **Channels with security in place.** Per-channel allowlists, confirm-before-acting on write actions, per-channel trust scoring (manual config in v0.4, learned in later versions).
- **Standing approvals.** User can convert a confirm-flow into auto-approve for a scope ("you can retry failed runs without asking"). Hard rails always still confirm.
- **Workflow templates and import/export.** Curated starter library + JSON import/export.
- **Custom tools — user-authored and meta-agent-generated.** Users write Python tools via an in-app code editor; registered tools appear in the agent config tool-picker alongside framework built-ins. The meta-agent gains a `generate_tool(description)` capability that produces a tool draft with an explicit user-approval review before it can run. Both paths share one file-backed registry surfaced via `GET /v1/tools` (added in v0.3) and the same per-tool sandbox isolation on execution.
- **Telemetry from raw Python scripts.** The framework's `TelemetrySink` protocol ships; `SprenTelemetrySink` (in `packages/spren/src/spren/telemetry/`) provides the Spren implementation; developers iterating with `python my_workflow.py` see runs appear in Spren.
- **Workflow pause / resume.** A user-initiated pause halts a running workflow with state preserved; resume picks up where it left off. Available on the run inspector UI, via REST endpoints, and as meta-agent tools (`pause_run` / `resume_run`). Paused runs survive daemon restart — on Spren startup, `Orchestra.list_paused_sessions()` populates the run-inspector "paused" tab. Snapshots older than 30 days are swept automatically; users can discard a paused run early via the inspector or the meta-agent. Resuming a snapshot whose `framework_version` does not match the running framework surfaces a clear error in the UI rather than silently corrupting state.

### Phase Y — TelemetrySink consumption + advanced linter (2 sessions)

| # | Title | Outcome |
|---|-------|---------|
| v0.4-27 | `SprenTelemetrySink` consuming the framework's `TelemetrySink` protocol | Spren ships one implementation of the framework's span-shaped `TelemetrySink` Protocol (one `publish_span` call per closed span; `publish_run_started` / `publish_run_completed` bookends; `publish_workflow` one-shot for new topologies). `SprenTelemetrySink` lives at `packages/spren/src/spren/telemetry/`. Wiring: `Orchestra(telemetry=[SprenTelemetrySink(endpoint=..., token=...)], ...)` — sinks register with the framework's `TraceCollector`, NOT directly with `EventBus`. The sink consumes spans whose `attrs` have already been filtered by the framework's `SecretRedactor`; raw passthrough requires the user to opt in to `NoRedactionRedactor`. Each payload carries `schema_version: int`; on mismatch the sink logs and graceful-degrades. The sink ships an HTTPS POST transport against the daemon's telemetry ingest endpoint (shape pinned in v0.4-27's brief) with async batching + a `shutdown()` that flushes pending spans within 5 seconds. End-user payoff: a `python my_workflow.py` script with `SPREN_ENDPOINT` + `SPREN_TOKEN` set produces visible runs in Spren's UI alongside meta-agent-spawned runs. Multi-consumer: the same protocol is satisfied by third-party `LangSmithTelemetrySink` / `PhoenixTelemetrySink` / `LangfuseTelemetrySink` packages — Spren is one consumer of a generic seam, not a special case. |
| v0.4-28 | Advanced semantic linter | Cost projection per workflow (using historical-mean), infinite-loop detection, rate-limit warnings, dependency-graph visualization |

### Phase Z — Workflow pause / resume (1 session)

| # | Title | Outcome |
|---|-------|---------|
| v0.4-29 | Workflow pause / resume (Spren-side, consumes framework Session 03) | New REST endpoints `POST /v1/runs/{id}/pause` and `POST /v1/runs/{id}/resume` calling into the framework's `Orchestra.pause_session()` / `Orchestra.resume_session()`. Spren constructs `Orchestra(storage_backend=FileStorageBackend(root=<data-dir>/data/runs))` and maps `run_id == session_id`, so the framework writes snapshots atomically to `<data-dir>/data/runs/{run_id}/snapshot.json`. The existing list endpoint with `?status=paused` is hydrated at daemon startup from `Orchestra.list_paused_sessions() -> list[PausedSessionMetadata]` so paused runs survive across `spren launch` cycles. An explicit discard surface (endpoint shape pinned in v0.4-29's brief — the simplest path is `POST /v1/runs/{id}/cancel` on a paused run calling `Orchestra.discard_paused_session(run_id)` immediately rather than waiting for the retention sweep) drops a snapshot before the 30-day sweep. Run inspector UI gains pause / resume buttons (visible on `running` and `paused` runs respectively); status badge gains a `paused` state; the inspector surfaces the snapshot age and a "discard" affordance for paused runs. The framework's periodic sweeper (invoked in `Orchestra.__init__`) deletes snapshots older than 30 days; Spren's `runs` row stays as run-history for inspection. `IncompatibleSnapshotError` (raised when `framework_version` differs between snapshot and running framework) is caught in the resume endpoint and returned as a 409 with a structured error payload (`code: "INCOMPATIBLE_SNAPSHOT"`); the inspector renders an explicit "snapshot was created on framework v{X}; running v{Y}; cannot resume" message and offers to discard. **At-least-once tool-call contract**: pause awaits the in-flight branch tick, but a tool call that was about to start when pause arrived re-runs on resume. Spren documents this in user-visible help; the workflow editor surfaces a warning at definition time when a workflow uses tools whose framework registry entry is non-idempotent. **Missed schedules during pause**: barrier timeouts and other scheduled events whose `fire_at` was crossed during the paused interval are skip-and-logged by the framework on resume; Spren's run inspector renders these as "missed during pause" entries in the trace view. Meta-agent gains `pause_run(run_id)` and `resume_run(run_id)` tools (suggest-with-confirm by default; pause is reversible so not hard-railed; resume on a previously-budget-exhausted run requires confirmation). The TUI surface gains the same pause / resume / discard controls (per SP-019, every client renders the same domain). |

## Hard constraints (carry over from v0.3)

All v0.3 hard constraints apply: no mocks, no backward-compat, no TRUNK-CRITICAL changes from Spren-side, full test coverage per session, framework-knows-nothing-of-Spren (SP-018), API-is-single-source-of-truth (SP-019).

**New v0.4-specific:**
- TUI client consumes the same FastAPI surface as the GUI (SP-019); zero TUI-specific endpoints.
- Tauri sidecar lifecycle (start, ready-detection, shutdown via stdin command, crash-restart) covered by integration tests.
- Auto-updater UI polish includes signature verification feedback before swap.
- Webhook payload signatures use HMAC-SHA256; secret rotation tested.
- Channel write operations are NEVER auto-confirmed; explicit user reply required.
- Sub-instance failure modes (timeout, crash, runaway cost) covered by integration tests.
- `TelemetrySink` is one of four framework features Spren motivates in v0.4 (alongside pause/resume completion, the workflow serializer, and the semantic linter). The protocol is span-shaped, registers via `TraceCollector`, and is multi-consumer by construction (LangSmith / Phoenix / Langfuse / Spren / generic-HTTP all satisfy it).
- **SP-023 enforcement:** every new always-on machinery module added in v0.4 (sub-instance lifecycle primitives, channel-adapter ABC, schedule primitives, etc.) MUST be split into generic-runtime files (no Spren-specific imports) and Spren-specific files (consume generic primitives via documented interfaces). Architect agents must identify the split for each module in the session brief; implementers verify by inspection at PR time. The boundary lives in code from v0.3 onward.

## End-of-v0.4-design conversation: package-shape decision

Before v0.4 implementation begins (i.e., when v0.4 architect-stage closes for all sessions), the architect agent MUST surface a strategic conversation with the user about the package-shape of the always-on runtime. Context: SP-023 codifies the boundary in code; the package-shape question (whether the generic runtime ships as a separate package, lives in `packages/spren/`, or motivates a rename of `packages/framework/` → `packages/marsys/` with internal split) was deferred from v0.3 because it depends on empirical evidence about alternative consumers — evidence v0.4 will start to produce.

**The trigger conversation prompt:**

> We agreed at the end of v0.3 to maintain the runtime/Spren boundary in code (SP-023) and defer the package-shape decision. Have we accumulated enough evidence to revisit?
>
> Three options to weigh:
> 1. Keep everything in `packages/spren/` with the boundary maintained in code only. Alternative consumers re-implement the surface OR vendor Spren as a dependency.
> 2. Extract `packages/marsys-runtime/` as a sibling package that Spren depends on. TUI / headless / third-party consumers can depend on `marsys-runtime` without taking `spren`.
> 3. Rename `packages/framework/` → `packages/marsys/`, with internal split: `marsys/workflows/` (current request-response framework), `marsys/runtime/` (extracted always-on). Then `packages/spren/` consumes both. Cleaner story for the marsys-as-product-line framing.
>
> Specific signals to bring to this conversation:
> - Are there alternative-consumer use cases that have surfaced (CLI/TUI users wanting headless deployment, third-party teams asking about the runtime, MARSYS Cloud / Studio architectural needs)?
> - How much code is now in the generic-runtime layer (size; module count)?
> - Are there observed pain points with the boundary-in-code-only approach (Spren-specific imports leaking into generic files; duplicated work; testing complexity)?
> - What changes in dependency direction would each option require?

The decision lands here, not at v0.5+. v0.5 ships against whichever package-shape v0.4 commits to.

Reference: [BRAINDUMP §13](../../../BRAINDUMP.md) "Always-on runtime: generic vs Spren-specific boundary maintained in code; package-shape deferred to end of v0.4" — the full reasoning that produced this position.

## Risks specific to v0.4

| Risk | Mitigation |
|---|---|
| **TUI development is parallel to GUI evolution** | Both consume the API; deviation is impossible by design (SP-019). New API endpoints land before client work. |
| **Tauri tray-icon UX divergence across platforms** | Use Tauri's cross-platform tray API exclusively; reject anything that requires per-OS branching beyond Tauri's documented surface. |
| **Cloudflare quick-tunnel URLs are ephemeral** | URL changes on tunnel restart. Surface in UI; persist as channel config. Offer named-tunnel mode for users with Cloudflare accounts (later release). |
| **Apple notarization adds release latency** | Notarization takes minutes-to-hour. CI handles async; release flagged "in notarization" until done. |
| **Telegram bot tokens reusable across instances** | Document that the bot can only be connected to one Spren install at a time; enforce on connect. |
| **Sub-instance runaway cost** | Per-instance budget cap; main agent kills if exceeded; surfaced in UI. |
| **Daily consolidation hallucinations** | Required provenance per fact; reject facts without `session_id:event_id` link to source event. |
| **Vector index stale after model swap** | Periodic full re-embedding plan documented; user-runnable `spren memory rebuild-index`. |
| **Channel trust scoring miscalibrated** | Manual config in v0.4; learned-from-usage deferred to later — false positives < silent corruption. |
| **`TelemetrySink` framework PR review delay** | Land Framework Session 02 before v0.4-27 starts; if blocked at cut time, push `SprenTelemetrySink` to v0.4.x patch — Spren's `python my_workflow.py` story degrades to "user runs Spren and uses the GUI" until the protocol ships. |
| **Pause/resume framework PR review delay** | The Spren v0.4-29 session depends on framework Session 03 being merged + released. Framework Session 03 is CRITICAL-tier with a mandatory ADR gate before any code is written, so review latency is structural rather than incidental. If blocked at v0.4 cut time, ship Spren v0.4 without pause/resume and follow up in v0.4.x. |
| **Pause-time partial state on disk corrupting on resume** | Framework Session 03 ships atomic writes via write-temp + `fsync(file_fd)` + `os.replace` + `fsync(parent_dir_fd)`; integration test in v0.4-29 covers paused-then-daemon-killed-then-restarted-then-resumed flow against `FileStorageBackend(root=<data-dir>/data/runs)`. |
| **Snapshot-version skew between Spren installs** | Framework rejects mismatched `framework_version` with `IncompatibleSnapshotError` (exact-string match in v0.4); Spren's resume endpoint returns 409 with a structured error code; inspector offers to discard. No silent migration in v0.4 — future framework PRs may add migration tooling once snapshot shape changes. |
| **Non-idempotent tools re-run on resume** | At-least-once contract is documented in Spren help; workflow editor surfaces a warning at definition time when a workflow uses tools whose framework registry entry is non-idempotent. Users opt in by acknowledging the warning. |
