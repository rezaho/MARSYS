# Framework features required for Spren v0.4

Spren v0.4 (TUI + Tauri polish + always-on org & channels) needs four framework features:

1. **`TelemetrySink` protocol + TraceCollector hook + `SecretRedactor`** — generic span-shaped observability hook the framework offers to any external backend; matches vendor SDK conventions (LangSmith, Phoenix, Langfuse). Spren ships one in-tree implementation; vendor adapters are contributor-welcome (none in-repo)
2. **Pause/resume completion in `state_manager`** — full reimplementation against the post-unified-barrier orchestrator; introduces the `StorageBackend` Protocol abstraction (`FileStorageBackend` ships; `S3` / `GCS` ship later via Cloud's PR); CRITICAL-tier with mandatory ADR escalation gate before any code is written
3. **Workflow definition serializer** — round-trip Pydantic ↔ runtime topology in `coordination/serialize.py`; re-exports `marsys.models.ModelConfig` (already Pydantic) and mirrors everything else; preserves `PatternConfig` via `Topology.metadata["original_pattern"]`; tool-name registry pattern for Agent.tools serialization
4. **Advanced semantic linter pass** — held until Spren Session 03 (the consumer) is drafted; consumer-driven design

Each lands as a separate framework PR. They can land in any order.

## Sessions

| 02 | [`TelemetrySink` protocol + TraceCollector hook + `SecretRedactor`](./sessions/v0.3.0/02-telemetry-sink-protocol.md) | Span-shaped `TelemetrySink` Protocol matching vendor SDK conventions (`publish_workflow` / `publish_run_started` / `publish_span` / `publish_run_completed` / `shutdown`, all `async`). ## Why these four together (not some now, some later)

Spren v0.4 is a substantial release; coordinating the framework features into a single Spren v0.4-aligned framework release window simplifies dependency management. Each feature is independent within the framework, so they can land in any order; the constraint is that all four be released before Spren v0.4 ships.

If one is delayed, Spren can degrade gracefully:

- **`TelemetrySink` delayed** → Spren v0.4 ships without `SprenTelemetrySink` (in `packages/spren/src/spren/telemetry/`); developers running raw Python scripts can't see their runs in Spren's UI until the framework PR lands; v0.4.x patch fills the gap. Spren's other surfaces (visual builder, meta-agent-spawned runs) are unaffected.
- **Pause/resume delayed** → Spren v0.4 session v0.4-29 (Spren-side endpoints + UI + meta-agent tools) cannot ship; user-facing pause / resume is pushed to v0.4.x patch or v0.5; running workflows remain cancelable but not pausable, and daemon restart still loses in-flight runs (the v0.3 behavior, regression-acceptable for v0.4 if needed)
- **Workflow serializer delayed** → Spren v0.4 keeps the AST-only Python-file import path from v0.3; the unified serializer waits
- **Semantic linter delayed** → Spren v0.4 keeps the v0.3 compile-time linter; the cost-projection / rate-limit warnings wait

## Multi-consumer notes

| Feature | Consumers beyond Spren |
|---------|------------------------|
| `TelemetrySink` | Spren (in-tree implementation); MARSYS Cloud's hosted observability (planned); LangSmith / Phoenix / Langfuse / OpenLLMetry adapters are contributor-welcome (the protocol shape matches their SDK conventions; no adapters shipped in this repo) |
| Pause/resume | MARSYS Cloud (managed long-runs need to survive node restarts), CI integrations (long workflows that span multiple jobs), framework users running locally who want crash resilience |
| Workflow serializer | Anyone wanting to persist a topology to disk; CI integrations loading topology JSON from a config file; community workflow templates / starter library |
| Semantic linter | Framework users iterating on workflows; MARSYS Cloud's pre-deploy validation; MARSYS Studio's authoring UI |

## Risks

| Risk | Mitigation |
|---|---|
| **`TelemetrySink` design churns mid-implementation** | Step 1 of Framework Session 02's brief is a design spike anchored on vendor SDK shape verification (LangSmith `create_run`, Phoenix OTel spans, Langfuse `start_observation`); the chosen span-shaped Protocol matches all three. The spike's findings are the design contract — substantive deviation from a vendor's signature requires escalation. |
| **Pause/resume requires touching `Orchestra` / `Orchestrator` (TRUNK-CRITICAL)** | Framework Session 03's brief opens with a mandatory ADR escalation gate. The implementer reads `coordination/orchestra.py`, `coordination/execution/orchestrator.py`, `coordination/state/state_manager.py` end-to-end before writing any code, drafts an ADR with six required sections, and gets written approval. Skipping the ADR returns the PR unread. |
| **Workflow serializer round-trips break under exotic patterns** | Property-based testing (Hypothesis) for round-trip; explicit test matrix for every `NodeKind` × `EdgeType` × pattern combination |
| **Semantic linter heuristics produce false positives** | Brief specifies that all new lint rules are `severity=warning`, never `severity=error`; framework users opt in to strict mode if they want |
| **Coordinated landing across four PRs vs. four separate framework releases** | Framework team's call; this dir doesn't prescribe |
