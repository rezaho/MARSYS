# Framework features required for Spren v0.5+

Looser plan. Items committed in scope but not locked in priority. Reprioritize when Spren v0.5 begins.

## Currently anticipated framework needs from Spren v0.5

Spren v0.5 brings tunnel-based messengers (Slack, WhatsApp), documented hosted deployment, code-generated tools with user approval, and possibly multi-user mode. The framework needs from these are mostly small and additive:

| Possible feature | Spren v0.5 driver | Multi-consumer justification |
|---|---|---|
| **Code-generation safe-execution sandbox** | Spren's "agent generates a Python tool function with user approval" feature needs a sandboxed execution context for the generated code | Any framework user running agent-generated tools; MARSYS Cloud's tool marketplace |
| **Tool capability scoping at the agent / workflow level** | Spren's per-workflow tool capability scoping (e.g., "this workflow may not use shell") | Framework users wanting per-workflow security boundaries; MARSYS Cloud's per-org policy enforcement |
| **Multi-tenant primitives** *(only if Spren v0.5 ships multi-user)* | Multi-user Spren needs framework hooks for tenant-scoped run history / memory boundaries | MARSYS Cloud (already has its own multi-tenant control plane, but the framework primitives would unify) |
| **OpenTelemetry GenAI semconv export** | Spren v0.5 may add a `--otel-export` for users who want traces in Langfuse / Phoenix / Datadog | Universal — every framework user who uses observability tooling |
| **`AgentSpec` agent-class discriminator + `pydantic_to_agents` dispatch** | Spren wants users to create-and-run specialized agents (Browser / Code / DataAnalysis / …) from the visual builder; today they cannot round-trip | Universal — ANY consumer that persists + rehydrates a `WorkflowDefinition` (MARSYS Cloud, Studio, third-party Python) silently downgrades a specialized agent to a base `Agent`. This is a framework serialization-fidelity gap, not a Spren UI concern |

These are not session-briefed yet. When Spren v0.5 begins, surface the actual needs, write briefs, and execute.

### Specialized-agent round-trip — concrete blocker (raised by Spren Session 08, 2026-05-17)

Spren's palette renders specialized agents **inactive** in v0.3/v0.4 (decided 2026-05-17) because the framework cannot serialize them faithfully — this is the real reason, recorded so it is not mistaken for a deferred Spren UI task (it supersedes the "specialized = a frontend preset" framing in the prior `AC-PALETTE-2` and in `docs/architecture/spren/11-node-model.md`).

Evidence (framework primary source, read this session — READ-ONLY from the Spren worktree):

- `AgentSpec` (`packages/framework/src/marsys/agents/serialize.py:50-68`, `ConfigDict(extra="forbid")`) mirrors only the **base** `Agent` constructor surface; there is **no** field naming the agent class.
- `pydantic_to_agents` (`serialize.py:158`) hard-codes `Agent(...)` — the base class — for every spec; no class dispatch exists.
- The specialized classes override real execution behavior, not just `__init__`: e.g. `InteractiveElementsAgent._run` (`agents/browser_agent.py:332`, the core loop), `BrowserAgent._pre_step_hook` (`:1969`), async construction (`create_safe` `:1085` / `_initialize_browser` `:1173`), resource lifecycle (`close`/`cleanup`/`__del__`), ~30 browser-control methods.

Consequently `BrowserAgent → AgentSpec → Agent` is lossy: a stored→materialized workflow reconstructs a base `Agent` with the same tools/instruction but none of the overridden behavior. The additive fix (fits backlog principles 1 + 2): an optional class discriminator on `AgentSpec` + a registry-based dispatch in `pydantic_to_agents`, plus a story for the specialized classes' async construction (`create_safe`). Scope it as its own framework session brief when picked up; do not bundle into a Spren-branch commit (principle 4).

## Strategic gates before drafting v0.5 framework support

- v0.4 framework features must be live and validated in the wild before drafting v0.5 framework needs
- Multi-user is gated on demand; if Spren v0.5 ships without multi-user, the framework's multi-tenant primitives are deferred too
- Code-generated tools require careful security review on both sides — sandbox design is a separate effort, not a bolt-on
